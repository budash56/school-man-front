# SchoolMan Frontend

SchoolMan Frontend is the staff dashboard for the SchoolMan school management system. It gives administrators, coordinators, registrars, and teachers a web interface for the daily work handled by the SchoolMan backend.

The app is built as an internal operations tool rather than a public website. Its screens are designed for repeated staff workflows: searching students, reviewing groups, managing planillas, recording school data, and navigating administrative tasks.

## What It Does

The frontend helps school staff:

- Log in securely and continue sessions with JWT authentication.
- Navigate role-specific school management areas from a dashboard shell.
- Manage students, enrollments, school years, class groups, classrooms, subjects, curriculum, users, attendance, and calendars.
- Review and edit planillas imported from spreadsheets or scanned from photos.
- Upload a photographed planilla to the scanner service through the backend.
- See OCR scan results, including detected student rows and review warnings.
- Work in light or dark mode with the preference saved locally.

## How It Fits In The System

SchoolMan is split into four repositories:

- `school-man-front`: React dashboard used in the browser.
- `school-man-back`: NestJS API and business-rule layer.
- `school-man-scanner`: FastAPI OCR service for photographed planillas.
- `school-man-deploy`: Docker Compose deployment wrapper.

The frontend does not talk directly to the database or scanner. It talks to the backend through `/api`; in local development Vite proxies `/api` to the NestJS server.

## Main User Workflows

Common workflows include:

- Coordinators enroll students and assign them to class groups.
- Administrators and coordinators manage school setup data such as years, groups, classrooms, users, subjects, and curriculum.
- Teachers review their assigned planillas and academic information.
- Staff upload a photographed planilla, wait for scanner results, then review detected rows.
- Registrars and coordinators use the frontend as the entry point for records and reports generated by the backend.

## Running Locally

Install dependencies:

```bash
npm install
```

Start the development server:

```bash
npm run dev
```

Open:

```text
http://localhost:5173
```

By default, local `/api` requests are proxied to the backend. If needed, set:

```dotenv
VITE_DEV_API_PROXY_TARGET=http://localhost:3000
VITE_API_BASE_URL=/api
```

## Validation

Useful checks:

```bash
npm run build
npm run lint
npm test
```

The production build may warn about bundle size because the app includes a large dashboard surface. That warning does not stop the build.

## Notes For Contributors

- Keep final authorization decisions in the backend.
- Keep pages task-focused; this is an operations dashboard, not a marketing site.
- Surface server errors clearly so staff can correct data or retry workflows.
- Treat OCR scan data as reviewable draft information, not final academic truth.
